## 5. 状态机图 (State Machine)

Lease & Password 生命周期状态转换

```mermaid
stateDiagram-v2
    [*] --> CONFIG_LOADED: Initialize

    state CONFIG_LOADED {
        [*] --> ReadConfig
        ReadConfig --> WriteKV: Config parsed
    }

    state WriteKV {
        [*] --> StoreInEtcd
        StoreInEtcd --> [*]: Success
    }

    CONFIG_LOADED --> ACTIVE_LEASE: Write to etcd

    state ACTIVE_LEASE {
        [*] --> Watching
        Watching --> [*]: Delete Event
    }

    ACTIVE_LEASE --> WATCH_MONITOR: Watch triggered

    state WATCH_MONITOR {
        [*] --> MonitorEvents
        MonitorEvents --> DeleteDetected: TTL expires
    }

    WATCH_MONITOR --> QUEUED: Publish to MQ

    state QUEUED {
        [*] --> InQueue
        InQueue --> Consumed: XReadGroup
    }

    QUEUED --> PROCESSING: Consumer reads

    state PROCESSING {
        [*] --> GeneratePass
        GeneratePass --> GetConfig: New password
        GetConfig --> StorePass: Config loaded
        StorePass --> SSHExecute: Password stored
    }

    PROCESSING --> SSH_EXECUTING: Execute SSH

    state SSH_EXECUTING {
        [*] --> Connecting
        Connecting --> Executing: Connected
        Executing --> [*]: Result
    }

    SSH_EXECUTING --> SUCCESS: Password changed
    SSH_EXECUTING --> FAILED: SSH error

    state SUCCESS {
        [*] --> UpdatePassword
        UpdatePassword --> [*]: Done
    }

    state FAILED {
        [*] --> RestorePassword
        RestorePassword --> [*]: Rollback
    }

    SUCCESS --> ACTIVE_LEASE: Recreate Lease
    FAILED --> ACTIVE_LEASE: Recreate Lease

    note right of ACTIVE_LEASE
        TTL: 30 seconds
        Key: <server>/lease
    end note

    note right of PROCESSING
        • GenPassword()
        • Get config from etcd
        • Store new password
    end note

    note right of SSH_EXECUTING
        • SSH connect
        • Execute passwd
        • Return result
    end note
```

**时间说明**：

- **33-35s**：SSH 执行密码修改
- **35-65s**：创建新 Lease，重启 30 秒循环

## 架构特点总结

### 1. 事件驱动架构
- 基于 etcd 的 lease 机制实现定时触发
- Watch 监听 key 删除事件
- Redis Stream + XGroup 实现消息队列

### 2. 分布式一致性
- 手动实现 2PC（两阶段提交）
- 预提交 + 回滚机制确保数据一致性
- 解决"密码修改后上游服务瘫痪"问题

### 3. 高可用设计
- etcd 集群（3 节点）
- Redis 集群（3 节点）
- Worker Pool（15 个 goroutine）
- 消费者组（2 个上游消费者 + 1 个下游消费者）

### 4. 30 秒循环机制
- Lease TTL: 30 秒
- 自动触发密码轮换
- 支持手动触发和定时触发

### 5. 安全隔离
- SSH 密码修改
- 独立的进程隔离
- 配置加密存储
